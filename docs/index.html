02. 먼저 알아야 할 기본 개념

이 프로젝트에서 계속 나오는 용어들입니다

Frontend (프론트엔드)

사용자가 직접 보고 클릭하는 화면. 이 프로젝트에서는 Next.js(TypeScript)로 만듭니다.

Backend (백엔드)

화면 뒤에서 데이터를 처리하는 서버. 화면에서 "신청" 버튼을 누르면, 백엔드가 그 데이터를 받아서 저장합니다. FastAPI로 만듭니다.

API (Application Programming Interface)

프론트엔드와 백엔드가 대화하는 방식. 예: 화면이 "신청 목록 줘"라고 요청하면, 백엔드가 데이터를 돌려줍니다. 이 요청/응답 규칙이 API입니다.

Database (데이터베이스, DB)

모든 데이터가 저장되는 곳. 엑셀 시트처럼 표(테이블) 형태로 데이터를 관리합니다. PostgreSQL을 사용합니다.

Docker (도커)

프로그램을 "상자"에 넣어서 어디서든 똑같이 실행되게 해주는 도구. 내 PC에서 되면 서버에서도 됩니다.

03. AI 관련 개념

이 프로젝트의 핵심인 AI 용어들입니다

Agent (에이전트)

스스로 판단하고 행동하는 AI. 단순히 질문에 답하는 게 아니라, 상황을 보고 → 필요한 도구를 골라서 → 실행까지 합니다.

LangGraph (랭그래프)

Agent의 판단 과정을 "흐름도(그래프)"로 만드는 도구. 접수 → 검색 → 판단 → 실행 같은 단계를 코드로 정의합니다.

RAG (Retrieval-Augmented Generation)

AI가 답변하기 전에 관련 문서를 먼저 검색하는 방식. "정책 문서가 100개인데, 이 신청과 관련된 정책만 찾아서 AI에게 보여주는 것"입니다.

MCP (Model Context Protocol)

AI가 외부 도구를 호출하는 표준 규격. Agent가 "권한을 승인해줘"라고 판단하면, MCP Tool이 실제로 DB에 승인 처리를 합니다.

LLM (Large Language Model)

ChatGPT, Claude 같은 대규모 언어 모델. 이 프로젝트에서는 Claude가 판단의 "두뇌" 역할을 합니다.

왜 이렇게 나누나? 화면, 서버, AI, 도구, DB를 각각 분리하면 한 부분을 고칠 때 다른 부분에 영향이 없습니다. 실무에서 프로그램을 만드는 표준 방식입니다.

데이터가 흘러가는 순서

순서누가뭘 하나
①Next.js (화면)"마케팅 대시보드 권한 주세요" 입력 → fetch로 FastAPI에 전달
②FastAPI (서버)요청 받음 → LangGraph Agent 실행
③Agent (접수)MCP Tool로 신청자가 누군지 조회
④Agent (검색)RAG로 관련 보안 정책 문서 검색
⑤Agent (판단)Claude(LLM)가 정책 보고 "팀장 승인 필요" 결정
⑥Agent (실행)MCP Tool로 팀장에게 알림 발송
⑦Agent (로깅)판단 과정 전체를 DB에 기록
⑧FastAPI → Next.js결과를 화면에 표시

05. 각 조각이 하는 일

Next.js
화면 (Frontend · TypeScript)
사용자가 보고 클릭하는 부분. 신청 폼, 채팅창, 승인 목록 등을 보여줍니다. fetch로 FastAPI에 요청을 보내서 데이터를 주고받습니다.
FastAPI
서버 (Backend)
화면에서 오는 요청을 받아서 처리합니다. "신청 저장해줘", "목록 줘", "AI한테 물어봐줘" 같은 요청의 진입점입니다.
LangGraph Agent
AI 두뇌 (LangGraph + Claude)
자연어를 이해하고, 정책을 검색하고, 판단하고, 도구를 호출합니다. 흐름도(그래프)에 따라 단계별로 실행됩니다.
RAG
문서 검색 (ChromaDB)
보안 정책 문서를 미리 저장해두고, 신청이 들어오면 관련된 정책만 골라서 Agent에게 넘깁니다.
MCP Server
AI의 도구 4개
Agent가 판단한 후 실제로 행동하는 도구들. 권한 신청, 조회, 승인, 알림 발송을 담당합니다.
PostgreSQL
데이터 저장소 (Database)
사용자, 권한 신청, 부여된 권한, AI 판단 이력 — 모든 데이터가 여기에 저장됩니다.

06. Agent 판단 흐름 (LangGraph)

Agent는 이 흐름도를 따라 단계별로 판단하고 행동합니다

① 신청 접수 — 신청자 정보 조회
↓
② 정책 검색 — RAG로 관련 보안 정책 찾기
↓
③ 판단 — 승인? 조건부? 에스컬레이션?
↓
Level 1 + 같은 부서 조건 충족 시 Level 2 이상
자동 승인 → 권한 부여
조건부 승인 → 조건 제안
에스컬레이션 → 담당자에게 전달
↓
④ 알림 발송
↓
⑤ 로그 기록 — 전체 판단 과정 저장

LangGraph란? 위 흐름도의 각 단계를 "노드(node)"라 부르고, 단계 간 연결을 "엣지(edge)"라 부릅니다. LangGraph는 이 흐름도를 코드로 만들어주는 도구입니다.

07. MCP Tool 4개 — AI의 도구

Agent가 판단한 후 실제 작업을 수행하는 도구들입니다

도구 이름하는 일예시
request_access 권한 신청 접수 "김철수가 고객 DB 접근을 신청했습니다" → DB에 저장
check_access 현재 권한 조회 "김철수가 지금 갖고 있는 권한이 뭐지?" → DB에서 조회
approve_access 승인/반려 처리 "승인 → 김철수에게 고객 DB 읽기 권한 부여" → DB에 저장
send_notification 알림 발송 "팀장님, 김철수의 권한 신청이 있습니다" → 알림

Agent ↔ MCP 연결 방식: Agent(LangGraph)는 MCP를 직접 호출하지 않습니다. MCP Tool을 LangGraph가 이해하는 형태로 감싸서(래핑) 등록하면, AI가 필요할 때 알아서 골라 씁니다.

08. DB 테이블 — 어떤 데이터를 저장하나

엑셀 시트 4개라고 생각하면 됩니다

users (사용자)
이름, 이메일, 부서, 역할
예: 김철수, 마케팅팀, 일반 사용자
access_requests (신청)
누가, 어떤 시스템에, 왜 신청했는지 + 현재 상태
예: 김철수, 고객DB, 마케팅 분석용, 대기중
permissions (권한)
현재 부여된 권한 목록 + 만료일
예: 김철수, 사내위키, Level 1, ~2026.12.31
audit_logs (판단 이력)
AI가 어떤 정책을 보고, 뭘 근거로 판단했는지
예: "고객DB 정책 참고 → Level 2 → 팀장 승인 라우팅"

11. 5주 로드맵

주차만드는 것끝나면 되는 것
1주차 FastAPI + DB + Next.js 화면에서 권한 신청 → DB 저장 → 목록 조회
2주차 MCP Server AI가 쓸 도구 4개 완성 + 관리자 승인 화면
3주차 RAG + LangGraph Agent 채팅으로 "권한 주세요" → AI가 정책 보고 자동 판단
4주차 통합 테스트 + 고도화 안정적으로 동작하는 데모
5주차 Docker + CI/CD + AWS URL로 누구나 접속 가능한 서비스 배포

매주 원칙: Backend에서 API를 만들고 → Frontend(Next.js)에서 fetch로 그 API를 호출해서 화면으로 확인합니다.
